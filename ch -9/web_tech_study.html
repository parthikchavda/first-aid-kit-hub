1 · ASP.NET Basics & Features
Core concepts, advantages, and architecture overview
Core
What is ASP.NET?
Definition, purpose, and origin
▾

Definition

ASP.NET (Active Server Pages .NET) is a server-side web application framework developed by Microsoft. It is the successor to Classic ASP (Active Server Pages) and is part of the .NET Framework.

  • Used to build dynamic websites, web apps, and web services.
  • Runs on the server; the browser only receives HTML output.
  • Supported languages: C#, VB.NET, J# (any .NET language).
⚡
Compiled (not interpreted)Code is compiled before execution → much faster than Classic ASP.
🧩
Language IndependentWrite in C#, VB.NET, or J# — all run on the same CLR.
🎯
Event-DrivenResponds to user actions (clicks, form submits) via server-side events.
🛡️
Built-in SecurityAuthentication & Authorization are built right into the framework.
Example – Simplest ASP.NET page
<%@ Page Language="C#" %>
<html>
<body>
  <% Response.Write("Hello, World!"); %>
</body>
</html>

What is CLR?

CLR (Common Language Runtime) is the execution environment for ASP.NET applications. It is the runtime component of the .NET Framework that manages the execution of .NET programs.

  • Provides memory management, garbage collection, security, and exception handling.
  • Converts MSIL code to native machine code via JIT (Just-In-Time) compilation.

5 · State Management

State
State Management Overview
HTTP is stateless — ASP.NET provides these mechanisms to persist data
▾

Why State Management?

HTTP is a stateless protocol — each request is independent. State Management addresses the problem of maintaining data across multiple requests from the same user.

MechanismScopeStorageUse Case
ViewStateSingle page (across postbacks)Hidden field in page HTMLPreserving control values on same page
SessionSingle user, all pagesServer memory (or DB)User login data, shopping cart
CookieSingle user, persistsClient (browser)Remember preferences, login token
QueryStringBetween pages via URLURL parameterPassing ID between pages
ApplicationAll users, whole appServer memorySite-wide counters, shared data
CacheServer-wideServer memoryExpensive database query results
ViewState Example
// Storing in ViewState
ViewState["UserName"] = "Alice";

// Retrieving
string name = ViewState["UserName"].ToString();

// Session Example
Session["CartCount"] = 5;
int count = (int)Session["CartCount"];
🔑
ViewStateMaintains control values across postbacks on same page. Stored as hidden field.
👤
SessionStores data for a single user across all pages. Server-side storage.
🍪
CookieStored in browser. Can persist after browser closes (persistent cookies).

6 · Caching & Performance
Output Caching, Data Caching, benefits and usage
Caching
Output Caching & Data Caching
How caching reduces server load and speeds up responses
▾

Output Caching

Purpose: Stores the rendered HTML output of a page. When the same page is requested again, the cached HTML is returned instead of re-running the server logic.

  • Dramatically reduces server processing for frequently requested pages.
  • Configured using the @OutputCache directive.
Output Caching Directive
<%@ OutputCache Duration="60" VaryByParam="None" %>
<!-- This page is cached for 60 seconds -->

Benefits of Caching

  • Faster page responses for users.
  • Reduced database and server CPU load.
  • Better scalability under high traffic.
🗄️
Output CacheStores rendered HTML. Reused for subsequent identical requests.
💡
Data CacheStores objects (like query results) in server memory for reuse.

Server Controls

Controls that are processed by the server (not browser-only). They have the runat="server" attribute and generate HTML output.

  • Examples: TextBox, Button, Label, GridView, AdRotator
  • AdRotator is a server control used to display rotating advertisements.

Validation Controls

Built-in controls to validate user input on both client and server side:

Validation ControlPurpose
RequiredFieldValidatorEnsures a field is not left empty
RangeValidatorChecks value is within a range
RegularExpressionValidatorValidates against a pattern (email, phone)
CompareValidatorCompares two control values
CustomValidatorCustom validation logic
ValidationSummaryDisplays all validation errors

User Controls vs Custom Controls

  • User Controls (.ascx) — Reusable page fragments created by drag-and-drop. Saved as .ascx files.
  • Custom Controls — User-defined controls built by inheriting from existing .NET classes. Compiled into DLLs.
Validation Control Example

<asp:RegularExpressionValidator 
  ControlToValidate="txtEmail"
  ValidationExpression="\w+@\w+\.\w+"
  ErrorMessage="Invalid email!"
  runat="server" />

9 · Master Pages & Themes

Master Pages
Create consistent headers, footers, and navigation across all pages

What are Master Pages?

Master Pages allow developers to create a consistent layout (template) across all pages of a website. They define the common structure (header, footer, nav) while Content Pages define the unique content for each page.

  • Master Page file extension: .master
  • Contains ContentPlaceHolder controls where child pages insert content.
Master Page Structure
// Site.master
<%@ Master Language="C#" %>
<html><body>
  <header>LOGO + NAVBAR</header>
  <asp:ContentPlaceHolder ID="MainContent" runat="server" />
  <footer>Copyright 2024</footer>
</body></html>

// Default.aspx (Content Page)
<%@ Page MasterPageFile="~/Site.master" %>
<asp:Content ContentPlaceHolderID="MainContent" runat="server">
  Welcome to the Home Page!
</asp:Content>

Themes

Themes provide a way to apply consistent visual styling (colors, fonts, skins) across all pages. Stored in the App_Themes folder.

📐
Master PagesConsistent LAYOUT (structure) — header, footer, nav shared across all pages.
🎨
ThemesConsistent VISUAL STYLE — colors, fonts, control appearances.
🧩
User ControlsReusable UI components (.ascx) — like a partial page fragment.

11 · Deployment
XCOPY deployment, Zero-touch, Publishing ASP.NET apps

XCOPY Deployment
Simple file-copy deployment — no Registry needed
▾

XCOPY / Zero-Touch Deployment

ASP.NET supports XCOPY deployment — simply copy application files to the server. No Windows Registry registration needed (unlike Classic ASP with COM components).

  • Also called "Zero-touch deployment" or "No-touch deployment".
  • Assemblies (DLLs) go in the Bin folder — no global registration needed.
  • Much simpler than Classic ASP deployment which required Windows Registry.

Client-Server Model

  • Client initiates the communication (sends requests).
  • Server receives and responds to requests.
  • Example: Browser (client) sends HTTP request → IIS (server) responds with HTML.

3-Tier / N-Tier Architecture

TierAlso CalledRoleExample
Tier 1Presentation Tier / Client Tier / UI TierUser interface — what user seesBrowser, HTML, ASPX pages
Tier 2Logic Tier / Business Tier / Middle TierBusiness rules and processingASP.NET code-behind, C# classes
Tier 3Data Tier / Database TierData storage and retrievalSQL Server, MySQL, Oracle

Server Types

Server TypeRoleExample
Web ServerHandles HTTP requests, serves static contentIIS, Apache, Nginx
Application ServerRuns business logic, processes dynamic requestsIIS (with ASP.NET), Tomcat
Database ServerStores and manages dataSQL Server, MySQL, Oracle
MCQ Tip — IIS

IIS (Internet Information Services) is an example of both a Web Server and an Application Server. It runs on Windows and hosts ASP.NET apps.

14 · Page Lifecycle & Events
Order of events in ASP.NET page execution
Lifecycle
ASP.NET Page Lifecycle Events
Order of execution from request to response
▾

Page Event Order (Important for MCQs)

ASP.NET Page Lifecycle Order
1. Page_PreInit       ← FIRST EVENT FIRED
2. Page_Init
3. Page_InitComplete
4. Page_PreLoad
5. Page_Load          ← Most commonly used
6. Control Events     (Button_Click, etc.)
7. Page_PreRender
8. Page_Render
9. Page_Unload        ← Last event (cleanup)
1️⃣
Page_PreInit — FIRSTFirst event fired when a page is requested. Used for themes, master pages.
📥
Page_LoadMost commonly used event. ViewState and PostBack data are available here.
🔚
Page_Unload — LASTCleanup — release resources, close connections.
🔁
PostBackWhen a page submits to itself (e.g., Button click). Check with IsPostBack.
IsPostBack Example
protected void Page_Load(object sender, EventArgs e)
{
    if (!IsPostBack)
    {
        // Runs only on first load, not on button clicks
        LoadData();
    }
}

Tracing

Tracing in ASP.NET is used to monitor execution and find errors. Enable with Trace="true" in the page directive or in Web.config. It shows page lifecycle events, ViewState size, control tree, and more.
